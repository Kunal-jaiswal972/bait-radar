# Deployment — Azure (Terraform)

The entire environment is defined as **Terraform** in [`infra/`](infra/) and deployed
with two scripts at the repo root:

- **`deploy.sh`** — provisions infra with Terraform, then builds + deploys all code.
- **`gen-local-settings.sh`** — pulls secrets from the deployed Key Vault into
  `local.settings.json` so local dev uses the same secrets as prod.

Everything lives in **one resource group, `BaitRadar`** (prod + local dev share it).
Dependencies are **not committed** — the scripts install them at deploy time.

## Architecture

```
   React SPA (web/) ─────────┐
   Azure Static Web Apps     │
                             ▼
   YouTube WebSub ──────▶ APIM (public front door) ──▶ NODE Function App
                                                        ├─ POST /api/channels
                                                        ├─ GET|POST /api/webhook/youtube
                                                        ├─ GET /api/dashboard/*
                                                        └─ EventHub trigger (worker)
                          Cosmos ◀── upsert                 │                 │
                                                            │ metadata/comments│ transcript (HTTP + key)
                          Event Hub ◀────────────────────── ┘                 ▼
                          (video-ingestion-hub)                       PYTHON transcript app
                                                                      (internal, youtube-transcript-api)
   Key Vault ── all secrets ── read by the Node app via managed identity
```

- The **Node** app is the only thing public (via APIM). The **Python transcript
  app** is internal — the Node worker calls it over HTTP with a function key.
- Transcripts can't be fetched from Node (YouTube gates the caption endpoint behind
  a session PoToken), so they live in a separate Python Function App.
- Every secret (Azure-generated connection strings/keys + the 3 non-Azure keys) is
  stored in **Key Vault**; the Node app reads them via `@Microsoft.KeyVault(...)`
  App Settings using its managed identity.

## 2. Configure

```bash
cd infra
cp terraform.tfvars.example terraform.tfvars   # public config
cp secret.tfvars.example    secret.tfvars      # 3 non-Azure secrets (git-ignored)
```

**`terraform.tfvars`** — set `subscription_id`; adjust regions/SKUs if needed. Defaults:

| Var | Default | Notes |
|---|---|---|
| `resource_group_name` | `BaitRadar` | single RG |
| `name_prefix` | `baitradar` | drives all resource names (no random suffix) |
| `location` | `centralindia` | most resources |
| `cosmos_location` | `westus2` | Cosmos (Students sub can't create it in Central India) |
| `ai_services_location` | `centralindia` | Vision + Language accounts |
| `static_web_app_location` | `eastasia` | SWA has no India region (CDN-served anyway) |
| `vision_sku` / `language_sku` | `S1` / `S` | Standard (only one free F0 per kind per sub) |

**`secret.tfvars`** — the only keys Azure can't generate (Terraform writes them to Key Vault):

```hcl
youtube_api_key           = "AIza..."
gemini_api_key            = "..."
pubsubhubbub_verify_token = "..."
```

> Everything else (Cosmos / Event Hub / Storage / Vision / Language keys) is
> **generated by Terraform** and pushed to Key Vault automatically.

## 3. Deploy

From the repo root:

```bash
bash deploy.sh
```

`deploy.sh` runs, in order:

1. `terraform init / plan / apply` (provisions the `BaitRadar` resource group).
2. **Node app** — `npm install`, `npm run build`, prune dev deps, `func azure functionapp publish` (ships `dist/` + `node_modules/`; deps installed fresh, not committed).
3. **Python transcript app** — `func azure functionapp publish` (remote Oryx `pip install` from `requirements.txt`).
4. Reads the Python app's function key → writes it to Key Vault (`transcript-function-key`) → restarts the Node app so it re-resolves the Key Vault references.
5. **Web SPA** — builds with `VITE_API_BASE_URL=<APIM>/api`, deploys `web/dist` to the Static Web App.

> If the SWA step fails with *"Could not find StaticSitesClient local binary"*, that's
> a transient CDN download — just re-run the last step (the CLI caches the binary).

At the end it prints the SPA / API / webhook URLs. Verify the API:

```bash
curl https://<apim>.azure-api.net/api/dashboard/channels     # -> []
```

## 3b. Redeploying after code changes

The infra is already provisioned, so day-to-day you only push code. All commands
are **idempotent** — re-running never recreates infra.

```bash
# Simplest: redeploy everything (terraform apply is a no-op when infra is unchanged)
bash deploy.sh

# Or target just what changed:

# Node app (APIs / webhook / ingestion worker / hourly tracker)
bun run build && func azure functionapp publish baitradar-app-api --javascript

# Python transcript app
cd transcript-service && func azure functionapp publish baitradar-transcript-api && cd ..

# Web SPA
cd web && VITE_API_BASE_URL=https://baitradar-apim.azure-api.net/api bun run build \
  && npx @azure/static-web-apps-cli deploy ./dist --env production \
     --deployment-token "$(terraform -chdir=../infra output -raw static_web_app_api_key)" \
  && cd ..
```

**Infra / config changes** (new Azure resource, app setting, SKU, region):

```bash
cd infra && terraform apply -var-file="secret.tfvars"
```

- **New non-secret setting** → add it to the `app_settings` map in `infra/main.tf`, `terraform apply`.
- **New secret** → add the value to `infra/secret.tfvars`, add its name to `secret_keys` in `infra/keyvault.tf`, reference it in the Node `app_settings` (`local.kv_ref["…"]`) in `main.tf`, then `terraform apply`. For a Python-app runtime secret, deploy.sh's key-wiring step is the pattern.

> After changing a **Key Vault reference** app setting, restart the Node app so it
> re-resolves: `az functionapp restart -g BaitRadar -n baitradar-app-api`.

## 4. Local development (from Key Vault)

Generate `local.settings.json` for both apps from the deployed Key Vault:

```bash
bash gen-local-settings.sh
```

This resolves secret **values** from Key Vault (KV *references* only work on the
Azure platform, not under a local `func start`) and sets
`TRANSCRIPT_FUNCTION_URL=http://localhost:7072/api/transcript`.

Run all three locally (3 terminals):

```bash
cd transcript-service && func start --port 7072   # 1. Python transcript app
bun run start                                      # 2. Node app (:7071)
cd web && bun run dev                              # 3. SPA (:5173, proxies /api → :7071)
```

**End-to-end pipeline test** (with the Node + Python hosts running):

```bash
# Atom body needs only yt:videoId + yt:channelId
curl -X POST http://localhost:7071/api/webhook/youtube \
  -H "Content-Type: application/atom+xml" --data-binary @upload.xml   # -> 202
curl http://localhost:7071/api/dashboard/videos/<videoId>            # -> 200 with detail
```

identity has *Get/List* on Key Vault.

## 6. Migrating off the old resource group

Once the new environment is verified end-to-end (SPA loads, a channel registers and
flips to `verified`, a video gets scored), decommission the old setup:

(Previously we were using yt-data-analyzer RG for that dismantling there are instructions below)

```bash
az group delete -n yt-data-analyzer --yes         # old RG + its resources
rm local.settings.json.bak-old-rg                  # backup of the old local config
```

Local dev and prod then both run entirely on `BaitRadar`.

## 7. Notes

- **State** is local (`infra/terraform.tfstate`, git-ignored) and contains secrets.
  For team use, migrate to a remote `azurerm` backend.
- **Key Vault** has `purge_protection_enabled = false` for easy teardown — set it
  `true` for real production.
- **APIM rate-limiting** is omitted (the `rate-limit-by-key` policy isn't supported
  on the Consumption tier). CORS for the SPA + `localhost:5173` is configured.
- **SPA deep links** work via `web/public/staticwebapp.config.json`
  (`navigationFallback` → `/index.html`).
- Destroy everything: `cd infra && terraform destroy -var-file=secret.tfvars`.
